# 从痛点到技能 - 思考框架

**文档目的**: 在创建技能前，通过系统化思考确保解决的是真实痛点，而非为了技术而技术

**适用场景**: Claude Code 技能创建、业务自动化需求分析、AI应用规划

**核心原则**: 痛点驱动 → 拆解分析 → 方案设计 → 评估决策

---

## 为什么从痛点出发？

### 常见误区

```
❌ 错误思维：
"我看到一个很酷的AI技术，看看能不能用到业务中"
"ChatGPT很火，我们应该用它做什么"
"别人都在用自动化，我们也做一个吧"

✅ 正确思维：
"渠道专员每天花3小时重复发布房源，太浪费时间"
"客户跟进不及时，经常丢单"
"数据统计靠手工，容易出错"
```

### 从痛点出发的优势

| 优势 | 说明 |
|------|------|
| **目标清晰** | 知道要解决什么问题，不迷失在技术细节 |
| **价值可衡量** | 痛点缓解程度 = 技能价值 |
| **成功率更高** | 解决真需求，而非伪需求 |
| **容易推广** | 团队看到实际好处，愿意使用 |

## 第一阶段：痛点识别

### 1.1 什么是真痛点？

**真痛点的特征**:
- 频繁发生（每天/每周）
- 耗时较长（30分钟以上）
- 容易出错（人工操作风险）
- 影响业绩（直接或间接）
- 有改进空间（非一次性任务）

**伪痛点示例**:
- "我想学习AI技术" → 这是学习需求，非业务痛点
- "这个系统界面不好看" → 可能是审美偏好，非核心痛点
- "偶尔有个任务很麻烦" → 频率太低，不值得自动化

### 1.2 痛点发现方法

**方法1: 时间记录法**
```
让团队成员记录一周工作：
- 每项任务耗时
- 任务频率
- 任务类型（重复性/创造性）

高频+高耗时 = 优先痛点
```

**方法2: 抱怨收集法**
```
收集团队常见抱怨：
"又要手工录入数据..."
"这个重复劳动太烦了..."
"又要加班做报表..."
"系统又出问题了..."

高频抱怨 = 痛点线索
```

**方法3: 流程观察法**
```
观察业务流程，找出：
- 多个系统切换
- 大量复制粘贴
- 手工核对数据
- 重复性操作

瓶颈环节 = 痛点所在
```

### 1.3 痛点描述模板

```yaml
痛点描述:
  业务场景: "渠道专员每天需要在多个平台发布房源信息"
  现状做法: "手工登录各平台，复制粘贴房源信息，调整格式"
  痛点表现:
    - 单次发布耗时: 15-20分钟/套
    - 每天发布量: 10-15套
    - 每日耗时: 2-3小时
    - 错误率: 约10%（格式错误、信息遗漏）
  影响范围:
    - 人员: 1名渠道专员
    - 频率: 每天
    - 业务: 房源曝光不及时影响获客
  期望结果: "自动或半自动完成多平台发布，节省时间，减少错误"
```

## 第二阶段：痛点拆解

### 2.1 拆解维度

**维度1: 按流程拆解**
```
完整流程: 数据准备 → 内容生成 → 平台发布 → 结果确认

拆解后:
├─ 数据准备: 房源信息从哪里来？格式如何？
├─ 内容生成: 需要什么格式？有哪些要求？
├─ 平台发布: 支持哪些平台？API是否开放？
└─ 结果确认: 如何知道发布成功？需要统计吗？
```

**维度2: 按角色拆解**
```
涉及角色:
├─ 渠道专员: 负责发布，直接受益
├─ 置业顾问: 提供房源信息，数据来源
├─ 店长: 关注发布效果，需要报表
└─ 客户: 看到房源信息，最终用户
```

**维度3: 按数据拆解**
```
数据流向:
房源数据 → 结构化处理 → 内容生成 → 发布记录 → 统计分析

数据类型:
├─ 房源基础信息: 标题、价格、面积、户型
├─ 房源描述: 文案、亮点、配套
├─ 媒体资源: 图片、视频
└─ 发布记录: 时间、平台、状态
```

**维度4: 按动作拆解**
```
高频动作:
├─ 获取房源信息
├─ 生成发布内容
├─ 登录平台
├─ 填写表单
├─ 上传图片
├─ 提交发布
└─ 记录结果

可自动化: 全部
需人工决策: 房源信息确认、发布时间选择
```

### 2.2 拆解工具

**工具1: 5Why分析法**
```
问题: 房源发布太慢

Why1: 为什么慢？ → 因为要逐个平台登录填写
Why2: 为什么逐个平台？ → 因为各平台格式不同
Why3: 为什么格式不同？ → 因为平台规则不一样
Why4: 能否统一处理？ → 可以，需要格式转换
Why5: 核心问题是什么？ → 缺乏统一的发布工具

根本原因: 缺少自动化发布机制
```

**工具2: 泳道图**
```
角色:    置业顾问    渠道专员        平台
           │          │              │
流程:   提供房源 → 整理信息 → 格式转换 → 发布
           │          │              │
系统:    内部表格    手工整理      手工填写
           │          │              │
自动化:   ───────────→──────→──────→
```

### 2.3 优先级排序

对拆解后的子问题进行排序：

```yaml
优先级评估矩阵:
  问题:
    - 任务: "多平台发布"
      频率: "每天10-15次"
      耗时: "2-3小时"
      价值: "直接影响获客"
      优先级: "P0（最高）"

    - 任务: "发布统计"
      频率: "每周1次"
      耗时: "30分钟"
      价值: "管理决策"
      优先级: "P1（高）"

    - 任务: "历史记录查询"
      频率: "偶尔"
      耗时: "10分钟"
      价值: "方便查询"
      优先级: "P2（中）"
```

## 第三阶段：方案对标分析

### 3.1 三维对标法

**维度1: 智能体平台能力对标**

| 平台能力 | 房产业务场景 | 是否需要 |
|---------|-------------|---------|
| 资讯收集生成 | 市场动态分析 | ✅ 已有 |
| 内容排版 | 房源描述优化 | ✅ 已有 |
| 客户服务对话 | 客户咨询接待 | ❌ 缺失 |
| 数据分析报表 | 业务数据统计 | ❌ 缺失 |
| 流程自动化 | 房源批量发布 | ❌ 缺失 |
| 销售支持 | 客户跟进提醒 | ❌ 缺失 |

**维度2: 员工岗位需求对标**

| 岗位 | 核心工作 | 痛点 | 对应技能 |
|------|---------|------|---------|
| 渠道专员 | 多平台房源发布 | 工作量大 | 房源发布管理器 |
| 置业顾问 | 客户跟进+房源匹配 | 跟进不及时 | 客户数据管理助手 |
| 店长/经理 | 绩效分析+决策 | 数据手工 | 数据统计分析助手 |
| 行政后勤 | 合同+档案+报表 | 重复繁琐 | 合同智能助手 |

**维度3: 重复动作提取**

| 重复动作 | 复用场景 | 抽象价值 |
|---------|---------|---------|
| 数据采集 | 房源/政策/竞品/客户 | ⭐⭐⭐⭐⭐ |
| 内容生成 | 文案/话术/报告 | ⭐⭐⭐⭐⭐ |
| 格式转换 | Markdown/HTML/小程序 | ⭐⭐⭐⭐ |
| 批量发送 | 微信/公众号/短信 | ⭐⭐⭐⭐ |
| 数据统计 | 绩效/业务/客户 | ⭐⭐⭐ |

### 3.2 交集分析

**高优先级交集**（三维度都有）：
```
房源发布管理器
├─ 平台对标: ✅ 流程自动化能力
├─ 岗位对标: ✅ 渠道专员核心工作
└─ 动作对标: ✅ 内容生成+格式转换+批量发布

投入产出: ★★★★★
学习难度: ★★☆☆☆
实施周期: 1-2周
```

### 4.2 MVP范围定义

**最小可行产品（MVP）原则**:
- 解决80%的核心问题
- 用20%的功能
- 快速验证价值

```yaml
房源发布管理器 MVP:
  第一阶段（核心功能）:
    - 支持单一平台（安居客）
    - 支持基础房源信息发布
    - 提供发布状态确认
    - 预计耗时: 1周
    - 验证目标: 自动化是否可行

  第二阶段（扩展功能）:
    - 扩展到2-3个平台
    - 支持批量发布
    - 添加基础统计
    - 预计耗时: 1周
    - 验证目标: 多平台处理能力

  第三阶段（优化功能）:
    - 智能发布时间优化
    - A/B测试不同文案
    - 与飞书多维表格集成
    - 预计耗时: 后续迭代
    - 验证目标: 持续优化

不在MVP范围:
  - ✗ 复杂的内容推荐算法
  - ✗ 全平台覆盖（先做主流平台）
  - ✗ 智能定价建议
  - ✗ 实时竞品对比
```

### 4.3 技术架构设计

```yaml
技能架构设计:
  目录结构:
    property-publisher-cskill/
    ├── SKILL.md
    ├── scripts/
    │   ├── collectors/      # 房源数据采集
    │   ├── generators/      # 内容生成（复用content-layout）
    │   ├── publishers/      # 多平台发布

    ├── config/
    │   ├── platforms.yaml   # 平台配置
    │   ├── templates.yaml   # 房源模板
    │   └── schedules.yaml   # 发布计划
    └── references/
        └── platform-api-docs/

  技术栈:
    - Python 3.9+
    - 现有content-layout技能（复用）
    - 平台API（安居客/链家/贝壳）
    - 飞书多维表格（数据源）

  数据流:
    房源数据 → 结构化处理 → 内容生成 → 平台发布 → 状态记录 → 统计分析
```

## 实战案例：房源发布管理器

### 完整思考过程

**阶段1: 痛点识别**

```yaml
发现过程:
  观察发现: "渠道专员每天花大量时间发布房源"
  深入了解:
    - 访谈对象: 渠道专员、店长
    - 工作观察: 跟踪一天工作流程
    - 数据收集: 记录发布耗时和频率
  痛点确认:
    - 任务: 多平台房源发布
    - 频率: 每天10-15套
    - 耗时: 2-3小时/天
    - 问题: 重复劳动，容易出错，影响其他工作
```

**阶段2: 痛点拆解**

```yaml
拆解分析:
  流程拆解:
    1. 获取房源信息（内部表格）
    2. 整理房源描述
    3. 登录各平台
    4. 填写表单（标题、价格、描述、图片）
    5. 提交发布
    6. 记录发布情况

  可自动化点:
    - ✅ 步骤1: 可从飞书表格读取
    - ✅ 步骤2: 可复用content-layout生成
    - ⚠️ 步骤3: 需要平台API或模拟登录
    - ✅ 步骤4: API可自动化
    - ✅ 步骤5: API可自动化
    - ✅ 步骤6: 自动记录到表格

  核心瓶颈:
    - 多平台切换（5-8个平台）
    - 格式不统一
    - 图片处理
    - 手工记录
```

**阶段3: 三维对标**

```yaml
对标分析:
  维度1 - 平台能力:
    ✅ 主流平台都有自动化发布案例
    ✅ 部分平台开放API
    ⚠️ 部分平台需要模拟操作

  维度2 - 员工需求:
    目标用户: 渠道专员（1人）
    痛点程度: 高（每天2-3小时）
    价值验证: 节省时间→专注获客

  维度3 - 公共动作:
    数据采集: ✅ 可复用
    内容生成: ✅ 已有content-layout
    格式转换: ✅ 可复用
    批量操作: ✅ 通用需求

  结论: 三维度交集，高价值+可复用
```

**阶段4: 评估设计**

```yaml
方案评估:
  业务价值:
    节省时间: 2-3小时/天
    错误减少: 约80%
    释放人力: 渠道专员可专注获客
    ROI: 非常高

  技术可行性:
    API支持: 主流平台支持
    复用性: 可复用content-layout
    学习曲线: 适合AI编程学习

  MVP定义:
    第一版:
      - 支持1个平台（安居客）
      - 基础发布功能
      - 简单状态确认
    验证: 技术可行性+用户接受度

  后续规划:
    - 扩展平台
    - 批量发布
    - 统计分析
    - 智能优化
```

**决策结果**: ✅ 批准实施，作为下一个创建技能

## 快速检查清单

创建技能前，问自己这10个问题：

### 痛点验证（3问）

- [ ] **Q1**: 这个痛点真实存在吗？有没有夸大？
  - 验证方法: 访谈用户、观察工作、记录数据

- [ ] **Q2**: 这个痛点发生的频率高吗？
  - 标准: 至少每周发生，最好是每天

- [ ] **Q3**: 解决这个痛点的价值大吗？
  - 评估: 节省时间、减少错误、提升业绩

### 可行性验证（3问）

- [ ] **Q4**: 技术上可行吗？
  - 检查: API是否可用、是否有类似案例

- [ ] **Q5**: 资源够吗？（时间、技能、预算）
  - 评估: 开发时间、维护成本、学习成本

- [ ] **Q6**: 有无可复用的现有技能？
  - 检查: 自己的技能库、开源项目

### 方案验证（4问）

- [ ] **Q7**: MVP范围定义清晰吗？
  - 标准: 能用20%功能解决80%问题

- [ ] **Q8**: 风险考虑到了吗？
  - 检查: 技术风险、业务风险、资源风险

- [ ] **Q9**: 有备选方案吗？
  - 准备: 如果技术不可行，Plan B是什么

- [ ] **Q10**: 成功标准明确吗？
  - 定义: 如何衡量技能是否成功
